## Root cause — `theheat-bot` red

The `test` job failed a single test:
`tests/test_scheduler.py::test_breaker_records_skipped_not_failed`
— `assert bot_state["source_health"]["slow_source"]["failed"] == 3` got `0`.

The rolling source-health window is anchored to the *latest* run
(`_source_health_window_cutoff` = `latest_ts − 7d`), **not** to wall-clock now.
The test seeded three timeout failures at fixed `2026-06-12T00/01/02:00Z`, then
let the circuit breaker record a `skipped` run stamped at *now*. Once real time
crossed 7 days past the seeds (today, 2026-06-19), the now-stamped skip slid the
window cutoff past the seeded failures and pruned all three → `failed == 0`.
Green every day until the calendar caught up. The sibling breaker tests
(`test_breaker_skips_after_3_timeouts`, `test_composite_runner_...`) shared the
fragile fixed dates but only assert breaker *status*, so they hadn't tripped yet.

## Fix (Mechanical)

Branch `fix/scheduler-breaker-test-time-drift`. Test-only change: added a
`_recent_timeout_timestamps()` helper that derives the seed timestamps from
`datetime.now(UTC)` (a few hours back, ascending, well inside the 7-day window),
and pointed all three breaker tests at it. **No product code changed** — the
rolling-window behavior is correct as-is.

Green gates (local): `pytest` 2022 passed / 40 deselected; `mypy src/` clean;
`ruff check .` clean. Dashboard untouched.
