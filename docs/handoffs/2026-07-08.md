# ★ Next build — #414 global heat records-cluster (spiked GO)

**Full spec:** [docs/superpowers/plans/2026-07-08-heat-records-cluster-spike.md](/Users/andrewpuschel/Documents/Claude/theheat/docs/superpowers/plans/2026-07-08-heat-records-cluster-spike.md).
Issue [#414](https://github.com/andrewzp/theheat/issues/414).

**Story:** "A heat dome pushed record highs in N cities across [region]" — the heat-dome's
many-records story, **global from day one** (rides the bot's already-worldwide GHCN +
Open-Meteo daily-record data; NO US-alert dependency — that's why it's on-brand where
row-13 was not).

**Why a new detection approach (not a patch):** a heat dome is a **spatial blob** that
ignores boundaries. Every existing clustering unit fails — country (fragments a European
dome across France/Belgium/NL/DE; Monaco/Luxembourg never cluster), reganom's 16 regions
(has `France`/`Iberia`/`UK & Ireland` but **no "Western Europe"** — a broad dome still
fragments), US states (US-only). So cluster **spatially**, then **name honestly**.

**Design (from the spike):**
1. **Data (one-line add):** the global daily-record station list already exists —
   `simultaneous_record_stations` at
   [src/orchestrator/sources/open_meteo.py:502](/Users/andrewpuschel/Documents/Claude/theheat/src/orchestrator/sources/open_meteo.py)
   (`{city, country, temp_c, kind:"high", old_record_c, old_record_year, margin_c,
   elevation_m, signal_date}`, from GHCN US + Open-Meteo world cities). **Add `lat`/`lon`**
   (known per city) so it can be clustered spatially.
2. **Cluster:** single-linkage by great-circle distance — cities within link distance **L
   (≈350 km, tunable)** chain into one cluster; minimum size **N (≈6, tunable)**.
   Deterministic (sort by (lat, lon, name); stable ids) — no run-to-run drift.
3. **Honest naming (the honesty gate — the crux):** tiered, first match wins:
   (a) cluster ⊆ one reganom `REGION_WATCHLIST` zone (≥~80% containment) → use that
   documented zone name; (b) else → **"N cities across {k} countries in {continent}"**,
   naming the top few countries by record count. Continent (coarse bbox lookup — small new
   util) and the country list are BOTH directly verifiable; **never coin a region name**
   ("Western Europe") that isn't a documented zone. This is what prevents mislabeling an
   Iberia+Maghreb cluster as "Europe."
4. **Bundle** `signal_kind="heat_records_cluster"` carrying `city_count`, `sample_cities`,
   `cluster_countries`, `cluster_continent`, `region_name` (or null), each a bundle fact
   the writer cites verbatim.
5. **Writer section + fact-check rule:** the writer leads with the cluster (not one city);
   a fact-check rule pins "only the carried geography labels — continent + country list;
   no region name beyond the carried `region_name`."
6. **Dedup:** the spatial cluster **supersedes** the flat `simultaneous_records` global
   count (retire/down-rank it for clustered events); **suppress the constituent individual
   *daily* drafts** for cities in a fired cluster (keep a standalone *all-time* record —
   bigger than the aggregate). Soft-dedup vs reganom on (region, date).
7. **State:** a per-cluster/date dedup key — a NEW `DEFAULT_STATE` key → **rides the
   sqlite persistence-contract test** (`src/state.py` + the contract test).
8. **Tiers:** v1 = daily records (data already collected). Monthly/all-time clustering is
   a natural v2 (collect those per-city records too; lead with the strongest tier).

**Open tuning (build-time, not blockers):** L and N (calibrate vs a few weeks of record-
station counts); the reganom-zone containment fraction; whether v1 includes monthly/
all-time tiers. **Existing landscape (don't rebuild):** `simultaneous_records` = global
same-day count + same-country roll-call (the wrong primitive — being superseded);
`country_record` = a country's single NATIONAL peak (different story); `regional_anomaly`
(reganom) = mean anomaly per curated region (different metric).

**This touches editorial + detection + state → codex-xhigh MANDATORY, looped to clean
APPROVE, LAST round after LAST edit. Likely a 2-PR split (PR-A detection+cluster+naming
foundation; PR-B writer/fact-check/dedup) — mirror how rows 11/14 split.**
