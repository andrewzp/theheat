## What shipped (all merged to `main` unless noted)

### The European-heatwave blind spot — root cause + the actual fix
The bot drafted **zero** heat about the late-June European heatwave (France 44.3°C /
hottest day ever, Spain 45.1°C, mounting deaths) while happily drafting US station
records and a Siberian fire. **Root cause:** `bot.yml` forced
`THEHEAT_SIGNALS_PROVIDER=ghcn` (the GHCN cutover, PR #33, **2026-05-05**). GHCN-Daily
has near-zero low-latency coverage outside the US, so the heat detector ingested
**0 European observations across its entire run history** — empirically **21/21 heat
events were US** (14 "United States" + 7 US-territory). Europe never entered the
pipeline. Non-US events still flowed from *other* global sources (fire/FIRMS,
dust/PM2.5, GPM precip), so the feed *looked* diverse and the gap was camouflaged.

- **[#332](https://github.com/andrewzp/theheat/pull/332) — `THEHEAT_SIGNALS_PROVIDER=both`** (`f89f525`, MERGED). New provider mode in `run_extreme_signals` (`src/orchestrator/sources/open_meteo.py`): run **both** GHCN + Open-Meteo and **partition by country** — US from GHCN (deep NOAA station records, unchanged), the rest of the world incl. Europe from Open-Meteo's curated cities. New pure module `src/orchestrator/signal_partition.py` (`is_us_location`, `partition_us_world`) — clean US/non-US split, no fragile name/geo matching. The two providers even label the US differently (`"US"` vs `"United States"`/territory brackets); the predicate handles both. Open-Meteo only fetches **non-US** cities in this mode, so the US half isn't slower. The downstream provider branches became **per-bundle** (`station_id` presence), provably identical for the single-provider modes — so US behavior is preserved exactly; this is purely additive coverage. `bot.yml` flipped `ghcn`→`both`. **Tradeoff:** US keeps deep all-time station records; non-US gets monthly/calendar/absolute-extreme records (Open-Meteo's archive is shallower than GHCN's). Antarctica is correctly NOT covered by the city heat detector (no cities) — its story is sea-ice/ice-mass, which the bot already tracks.

### Heat coverage watch — the monitoring class-fix
The deeper lesson from the five-whys: **monitoring measured whether sources *work*,
never whether coverage matches the global mission.** A continent-blind detector read
"green" for 7 weeks. The fix makes that unrepresentable.

- **[#333](https://github.com/andrewzp/theheat/pull/333) — heat coverage watch** (`444be3f`, MERGED). Records each surfaced **hot** heat event's geography **at the source** into a **persistent rolling tally**, and opens an advisory auto-closing GitHub issue (sibling to the yield-watch) when one country **or** continent dominates ≥85% of heat over ≥20 events in a 21-day window — or alarms if the bot is drafting but coverage data is missing. Built v1 = **heat only** (the one class with clean per-event `country`); other classes deferred to per-source instrumentation.
  - **Record at source:** `state.record_coverage_observation(...)` is called at the 3 heat enqueue sites in `run_extreme_signals` (per-city, record-streak, country-record), **hot-gated** (cold extremes excluded). Continent resolved once at write time (`src/coverage.py` `resolve_continent`, backed by `data/country_continent.json` derived from cities.csv by `scripts/build_country_continent.py`, with hot-region corrections e.g. Saudi Arabia→Asia). Best-effort — a recording failure can never break a draft.
  - **Persistent `coverage_log`** in state: new key wired through `DEFAULT_STATE` + `MERGE_SPEC` (dedup-on-`event_id`) + `BotState`/`CoverageRecord` (`src/state_schema.py`) + SQLite `_METADATA_JSON_KEYS` (`src/storage/sqlite_store.py`) — decoupled from the 20-run `run_history` cap (the v1-design flaw codex caught: the cap = a <1.5-day window, far too short for a 20-event threshold). Pruned to 21 days.
  - **Pure `coverage_watch` classifier** in `scripts/source_health_sentinel.py`, mirrored byte-for-byte in `dashboard/lib/source-health.js`. Three finding kinds: `mono_regional` / `insufficient_data` (5–19 events — visible, **never silently green**) / `no_data` (<5 while drafting). Reconciled into one advisory issue (`COVERAGE_WATCH_TITLE`/`_MARKER`), `insufficient_data` never opens an issue. Constants (identical both sides): `COVERAGE_WINDOW_DAYS=21`, `COVERAGE_MIN_EVENTS=20`, `COVERAGE_CONCENTRATION=0.85`, `COVERAGE_DATA_FLOOR=5`, `COVERAGE_WATCHED_CLASSES=("heat",)`.
  - **Built via subagent-driven development** (7 tasks, fresh implementer + reviewer per task) off a plan that was **reviewed twice by codex** (round 1 caught 4 P1s: state_schema parity, SQLite persistence, real issue lookup, insufficient-data-not-silent; round 2 caught 3 more: cold-as-heat, ruff F841, render miswiring). A final **Opus whole-branch review** executed the actual-incident shape (22 events, ~100% US) through both the Python and JS implementations and confirmed the blind spot is caught end-to-end (country axis, share 1.0). Design + plan: `docs/superpowers/specs/2026-06-25-coverage-monitor-design.md`, `docs/superpowers/plans/2026-06-25-heat-coverage-watch.md`.
  - **Activation is passive:** the next bot cron starts writing `coverage_log`; the hourly sentinel starts running the check. It needs ~20 hot events over 21 days before it can render a verdict — until then it reports `insufficient_data` (visible, not alarming, not silent). **Dashboard UI render is deferred** (the `coverage` payload is exposed in `buildSourceHealthPayload`; the GitHub issue is the primary surface). That render is a chipped follow-up.
