## The source-health sentinel (the session's main build — how it works)

- **What:** `scripts/source_health_sentinel.py` (pure, unit-tested classifier + `plan_issue_actions`) + `.github/workflows/source-health-sentinel.yml` (cron `30 */4 * * *`, every 4h after each bot cycle).
- **Behavior:** for every source it computes a category — `healthy` / `degraded` / `idle` / `failing`. A `failing` source (recent active success rate <50%) gets a GitHub issue `Source down: <name>`, labeled `source-health-sentinel`, assigned to `andrewzp`, with a body that states the **cause** (`ours` = code/auth/moved-endpoint → fix it; `external` = NASA/gov down → find an alternate feed) and the fix. When the source recovers, the issue **auto-closes**. Upstream-only days mutate nothing.
- **Idle rule:** judged over the recent *run* window (skips included). All-skip window → `idle` (a cadence source between runs isn't "failing" on stale attempts). A *recent active* attempt that failed still → `failing`.
- **Local use:** `python scripts/source_health_sentinel.py /tmp/state.json` is a safe **dry-run** (prints the verdict, mutates nothing). It runs live only in CI (`GITHUB_ACTIONS=true`) or with `--apply`.
- **The dashboard mirrors this** (0.9.14.0): same `external`/`idle` semantics so the panel and the issues tell one story. **If you change the Python `classify_error`, change the JS `classifyError` in `dashboard/lib/source-health.js` to match** (two implementations, kept in sync by convention).
