## Problem Statement

25% of all climate tweets come from bots, and almost all of them spread disinformation. Pro-truth automated climate accounts barely exist (~5% of bot climate tweets). Meanwhile, @extremetemps (the closest thing to a climate data account) is manually run by one person with no automation and no personality. There is no @earthquakeBot equivalent for heat and climate events. The automated truth side of climate Twitter is wide open.

The user owns @theheat on Twitter/X and wants to fill this gap with a fully automatic, personality-driven climate bot that feels like breaking news, not a NOAA press release.

## What Makes This Cool

The core mechanic is the **"Hot 10" daily leaderboard** — cities ranked not by absolute temperature but by how much hotter they are versus historical norms. Phoenix is always hot, nobody cares. But "Miami just knocked Baghdad off the #1 spot" creates city rivalry, screenshots, and a daily habit loop. The leaderboard turns passive climate data into a competitive spectacle.

Supporting content streams (heat records, wildfires, CO2 milestones, Arctic ice, streak tracking) keep the feed dynamic between daily leaderboard posts.

The personality is the product. Dark humor, dry wit, urgency. "Phoenix. Again. 119 degrees F. That's a new record. The old one was set... last year." The data is free and public. The voice is why people follow.

## Constraints

- **$0/month operating cost.** All infrastructure must be free tier.
- **Fully automatic.** No manual intervention after deployment.
- **No Mac Mini dependency.** Cloud-only architecture.
- **No paid X API tier at launch.** Free tier (1,500 tweets/month posting). Reply bot deferred.
- **Revenue goal: profitable.** Not a charity project. Monetization designed in from day one.

## Premises

1. **Gemini Flash API (free tier, 1,500 req/day)** generates personality-driven tweets. Quality is sufficient for short-form content. Prompt engineering is the real product work.
2. **Multiple free data sources** (NOAA CDO, NASA FIRMS, Mauna Loa CO2, sea surface temps, ERA5 reanalysis for climatological normals) provide differentiated, multi-signal content.
3. **X API free tier** for posting (1,500 tweets/month). Reply bot requires paid access + explicit written X approval for AI replies, deferred to Phase 2.
4. **Revenue path: X Creator Revenue** (ad sharing at 500 followers + 5M impressions), **newsletter** (Beehiiv/Substack with ad monetization), and **sponsorships** from green/climate brands at scale.
5. **@theheat voice** (dark humor, urgency, dry wit) is the differentiator. Data is fuel. Personality is why people follow.
6. **Fully cloud-based:** GitHub Actions (free, public repo) + Gemini Flash API (free) + X API (free). No server.

## Approaches Considered

### Approach A: "The Wire" — Minimal Viable Bot
Single-stream heat record bot. NOAA data to Gemini Flash to X post. Ships in a weekend. Validates voice and audience before building more. $0/month. Rejected because it lacks the viral mechanic (leaderboard) and monetization infrastructure (newsletter).

### Approach B: "The Scoreboard" — Hot 10 + Multi-Stream (CHOSEN)
Daily "Hot 10" anomaly leaderboard as anchor content. Heat records, wildfires, and CO2 as supporting streams. Newsletter + landing page from day one. Cross-posts to Bluesky/Threads. Ships in 1-2 weeks.

### Approach C: "The Network" — Platform Play
Full media brand. X + Bluesky + Threads + newsletter + website dashboard + merch. 3-4 weeks before first tweet. Rejected because it over-engineers before validating the content and voice.

## Recommended Approach

**Approach B: The Scoreboard**

### Architecture

```
GitHub Actions (two cron schedules)
  |
  |-- Daily leaderboard job (runs at 8:00-8:30 AM ET, 12:00 UTC)
  |     |-- Fetch current temps for 150 cities via Open-Meteo (free, no auth)
  |     |-- Compare against pre-computed climatological normals (static CSV in repo)
  |     |-- Rank top 10 by anomaly (current minus normal)
  |     |-- Generate Hot 10 tweet via Gemini Flash
  |     |-- Post to X + Bluesky
  |
  |-- Alert polling job (runs every 4 hours, 6x/day)
  |     |-- Fetch NOAA CDO daily records (broken/tied, last 72 hours)
  |     |-- Fetch NASA FIRMS active fire data (high-confidence large fires)
  |     |-- Fetch Mauna Loa CO2 daily reading
  |     |-- Deduplicate against state.json (previously posted events)
  |     |-- If new events found: generate tweet via Gemini Flash, post
  |     |-- If API errors: log, skip this source, retry next cycle
  |
  v
Gemini Flash API (free tier)
  |-- Generate tweet with @theheat voice from structured data prompt
  |-- Voice calibrated via VOICE.md (personality spec, examples, banned phrases)
  |-- If Gemini fails: fall back to template-based tweet (no AI, still posts)
  |
  v
Post to X API (free tier, 1,500/month)
  |-- Cross-post identical content to Bluesky via AT Protocol (free, same text)
  |
  v
Commit updated state.json to repo (dedup state)
  |-- GitHub Actions concurrency group ensures no overlapping commits
```

**Rate limit budget (worst case):**

| Resource | Limit | Daily usage (worst case) | Monthly usage |
|----------|-------|--------------------------|---------------|
| X API tweets | 1,500/month | 8 tweets | 240/month (16% of limit) |
| Gemini Flash | 1,500/day | 10 requests | 300/month (0.7% of limit) |
| Open-Meteo | Unlimited (fair use) | 1 request (150 cities) | 30/month |
| NOAA CDO | 1,000/day | 6 requests | 180/month |

**Error handling:** Each data source fetch is wrapped in try/catch. If a source fails, that content type is skipped for this cycle and retried on the next run. Failures are logged to a `errors.json` file committed alongside state.json. If Gemini fails, the bot falls back to a template-based tweet using the raw data (still posts, just without personality).

### Content Calendar (daily)

| Time | Content Type | Source | Schedule |
|------|-------------|--------|----------|
| ~8 AM ET | Hot 10 Daily Leaderboard | Open-Meteo + static normals | Daily cron (12:00 UTC) |
| Every 4 hours | Heat Record Alerts (if any) | NOAA CDO daily records | Alert polling cron |
| Every 4 hours | Wildfire Alerts (if any) | NASA FIRMS (high FRP fires) | Alert polling cron |
| Every 4 hours | CO2 Milestones (if threshold hit) | NOAA GML Mauna Loa | Alert polling cron |
| Weekly (Sunday) | Week in Review thread | All data, streak updates | Weekly cron |

Note: GitHub Actions cron may drift 5-30 minutes during peak load. The "~8 AM ET" leaderboard post may arrive between 8:00-8:30 AM. This is acceptable for the use case.

### Voice Specification (VOICE.md)

**Character:** @theheat is a cynical weatherman who's seen too much. Dry, dark, occasionally funny. Never preachy. Never uses hashtags or emojis. Sounds exhausted by the data, not angry about politics.

**Seed examples (illustrative, not final. The 30-tweet golden set is a separate deliverable built during Step 2 of Next Steps):**
- "Phoenix. Again. 119 degrees F. New record. The old one was set... last year."
- "Day 47 above 110 in Phoenix. At this point the streak has its own personality."
- "Congratulations to Miami for making the Hot 10 for the first time. Nobody asked for this."
- "CO2 hit 428 ppm today. For context, it was 280 ppm for the entire history of human civilization. We're speed-running this."
- "New wildfire detected in Northern California. Satellite confidence: HIGH. 0% contained. It's April."

**Banned patterns:** No emojis. No hashtags. No "BREAKING:" prefix. No climate policy opinions. No "we need to act now" messaging. No exclamation points. Let the data be the outrage.

**Tone knobs:** 70% dry observation, 20% dark humor, 10% genuine awe at the data.

### Data Sources (all free, no auth or free auth)

| Source | Data | Freshness | Auth | Notes |
|--------|------|-----------|------|-------|
| Open-Meteo API | Current day high temps for 150 cities globally | Real-time | None (free, no auth) | Primary source for Hot 10 leaderboard |
| NOAA CDO API | Daily temp records broken/tied (US stations) | 24-72 hours | Free token | Heat record alerts |
| NASA FIRMS | Active fire detections globally | Near real-time (3hr) | Free account | Wildfire alerts |
| NOAA GML | Mauna Loa CO2 daily | Daily | None (public CSV) | CO2 milestone posts |
| NSIDC | Arctic sea ice extent | Daily | None (public data) | Ice tracking (future) |
| Static normals CSV | 30-year climatological normals per city/month | Pre-computed once | N/A (in repo) | For Hot 10 anomaly calculation |

**Note on climatological normals:** ERA5 reanalysis data requires batch processing of multi-GB NetCDF files, which won't run in a GitHub Actions environment. Instead, normals are pre-computed once (offline or locally) from ERA5 or NOAA Climate Normals data, saved as a simple CSV lookup table (city, month, avg_high_temp), and committed to the repo. The bot reads this static file at runtime. This CSV only needs updating every few years.

### Hot 10 Leaderboard Calculation

**City list:** 150 cities curated by global population + geographic diversity. Stored as `cities.csv` in the repo (columns: city_name, country, lat, lon, open_meteo_id). Includes the 100 most populous cities plus 50 additional cities chosen for geographic coverage (Arctic, island nations, Southern Hemisphere, etc.).

**Calculation steps:**
1. Fetch current day's high temperatures for all 150 cities from Open-Meteo Forecast API (150 individual requests parallelized, ~2 seconds total, free, no auth, global coverage)
2. Look up 30-year monthly climatological normal for each city from `normals.csv` (static file in repo, pre-computed once from ERA5 or NOAA Climate Normals)
3. Compute anomaly: actual_high minus normal_high for this month
4. Rank top 10 by anomaly (not absolute temperature)
5. Compare against yesterday's Hot 10 from state.json: track position changes ("Miami UP 3 spots, Baghdad DOWN 2")
6. Track streaks: consecutive days a city has appeared in the Hot 10 (stored in state.json)
7. Format as a single tweet or 2-tweet thread if needed
8. Enforce daily tweet cap: max 10 tweets/day tracked in state.json as a safeguard against API limit overruns

**state.json schema (key fields):**
```json
{
  "last_hot10": {"date": "2026-04-07", "cities": ["Miami", "Baghdad", ...]},
  "streaks": {"Miami": {"consecutive_days": 14, "last_seen": "2026-04-07"}, ...},
  "posted_events": ["noaa_record_PHX_20260407", "firms_fire_12345", ...],
  "daily_tweet_count": {"2026-04-07": 5},
  "errors": [{"source": "noaa", "ts": "...", "msg": "..."}]
}
```

**NOAA record detection:** NOAA CDO does not expose a "broken records" endpoint directly. The bot queries GHCND daily observations for monitored US stations, compares against historical extremes for that calendar date (also from GHCND), and flags new records. Alternative: use NOAA's ACIS (Applied Climate Information System) RCC web services, which expose pre-computed records. Implementation should evaluate both approaches during build.

**Fire alert thresholds:** NASA FIRMS events are filtered to confidence >= 80% AND Fire Radiative Power (FRP) >= 100 MW. This eliminates small agricultural burns and low-confidence detections.

**CO2 milestone triggers:** Post when daily Mauna Loa CO2 reading crosses a new integer PPM for the first time (e.g., first reading at 429 ppm). Also post weekly comparison: this week's average vs. same week last year.

**Concurrency:** Both the daily leaderboard and alert polling jobs run in a single GitHub Actions workflow file with a shared concurrency group `theheat-bot`, ensuring no overlapping runs.

## Eng Review Decisions (2026-04-07)

Resolved during /plan-eng-review:

1. **State storage:** GitHub Gist (not git commits). One API call to read, one to write. Same GitHub token. No branch pollution.
2. **HTTP requests:** Sequential (no async). 150 Open-Meteo calls take ~45 seconds. Runs once/day in a cron job. Nobody is waiting.
3. **Record detection:** Open-Meteo detects likely records in real-time. NOAA ACIS confirms 24-72 hours later. Two-part content pattern: "appears to have broken" → "NOAA confirms."
4. **Content moderation:** Two-layer safety. Layer 1: deterministic regex gate (emojis, hashtags, banned phrases). Layer 2: Gemini safety check for contextual tone. 3x retry on failure, then template fallback.
5. **Build order:** Phase 1A = alerts + voice validation (ship in ~3-4 days). Phase 1B = Hot 10 leaderboard (add after voice is validated). Decouple to test the riskiest assumption (voice) before building the hardest feature (leaderboard).
6. **Test suite:** Full pytest coverage. Mocked API tests, pure logic unit tests, eval tests for voice quality. 38 codepaths identified.
7. **NOAA CDO:** Dropped entirely. Replaced by Open-Meteo historical (global) + NOAA ACIS (US authority).
8. **Critical gap:** Normals CSV needs validation. Add sanity check: anomaly > 30C flags as likely data error, skip city.
